# 📚 ארכיטקטורת המערכת (System Architecture & Interview Guide)

מסמך זה מרכז את כל השיקולים ההנדסיים, הארכיטקטורה, והניתוח הטכנולוגי של מערכת **Event-Driven AI Developer Agent**.

---

## 🎯 1. מטרת המערכת (System Goal)

המערכת היא **AI Developer Agent** המבוססת על ארכיטקטורת **Hybrid RAG** (Retrieval-Augmented Generation) ומנוהלת ע"י אירועים.

### 🔴 הבעיה שהמערכת פותרת
מערכות RAG קלאסיות (Standard RAG) סובלות ממגבלה עיקרית: הן מניחות שכל שאילתות המשתמש דורשות דמיון וקטורי בלבד. 
* **שאילתות מובנות** (למשל: "מהן כל ההחלטות שנלקחו בפרויקט?") דורשות שליפה מלאה ומדויקת מתוך מאגר מובנה, ומעבר דרך חיפוש וקטורי עלול להוביל לחסר או לשליפה חלקית.
* **שאילתות פתוחות** (למשל: "איך מיושם ניהול המצב?") דורשות חיפוש סמנטי עמוק מתוך תיעוד טקסטואלי חופשי.

### 🟢 הפתרון ההנדסי
ניתוב דינמי (**Routing**) שמזהה את כוונת השאילתה ומנתב אותה באופן אוטומטי:
1. **מסלול נתונים מובנים (Structured Extraction Path):** שליפה ישירה מקובץ JSON מובנה וניסוח מעובד על ידי ה-LLM.
2. **מסלול חיפוש סמנטי (Semantic Vector Search Path):** שליפה וקטורית מ-Pinecone, דירוג מחדש (Reranking) וסינון איכותי למניעת הזיות (Hallucinations) לפני העברה למודל השפה.

## 🛠️ 2. טכנולוגיות, ספריות ושיקולים הנדסיים

### 1. LlamaIndex & LlamaIndex Workflows
* **מה זה?** Framework מתקדם לבניית יישומי LLM, RAG וסוכנים אסינכרוניים.
* **במה השתמשנו?** ב-**LlamaIndex Workflows** המבוסס על **Event-Driven Architecture**.
* **למה דווקא זה?** בשונה משרשראות ליניאריות (כמו Chains בסיסיים), Workflows מאפשרים בניית מכונת מצבים (State Machine). כל שלב (`Step`) הוא פונקציה אסינכרונית עצמאית שמקבלת אירוע (`Event`) ומחזירה אירוע חדש. זה מאפשר הסתעפויות (Routing), טיפול במצבי קצה, ותחזוקה קלה.

### 2. Cohere API
* **מה זה?** פלטפורמת AI המציעה מודלי שפה, מודלי הטרמה (Embeddings) ומודלי Reranking.
* **במה השתמשנו בפרויקט?**
  * **LLM (`command-r-08-2024`):** מודל שפה המותאם למשימות RAG, ניסוח תשובות מדויקות ועבודה עם קונטקסט.
  * **Embedding Model (`embed-multilingual-v3.0`):** מודל הווקטוריזציה שהמיר את התיעוד למערכים מתמטיים (Vectors). נבחר בשל התמיכה המצויינת שלו בריבוי שפות (עברית ואנגלית).
  * **Reranker (`CohereRerank`):** מודל Cross-Encoder שמבצע דירוג מחדש למסמכים שנשלפו לפי מידת הזיקה שלהם לשאילתה, מה שמשפר משמעותית את דיוק ה-Context שנכנס ל-LLM.

### 3. Pinecone Vector Database
* **מה זה?** מסד נתונים וקטורי בענן (Vector DB).
* **במה השתמשנו ולמה?** אחסון ושליפה של ה-Embeddings של מסמכי התיעוד (`.md`).
* **למה דווקא Pinecone?** מאפשר חיפוש דמיון וקטורי (Similarity Search) במהירות גבוהה (Low Latency) וזמינות גבוהה בענן.

### 4. Gradio
* **מה זה?** ספרייה ליצירת ממשקי משתמש (UI) מהירים ב-Python.
* **במה השתמשנו?** בניית ממשק צ'אט אינטראקטיבי המחובר בצורה אסינכרונית להרצת ה-Workflow.

### 5. Python-Dotenv & Environment Variables
* **במה השתמשנו ולמה?** ניהול מאובטח של מפתחות API ופרמטרים של השרת מחוץ לקוד המקור.

## 🧱 3. פירוק השלבים (Steps) והאירועים (Events) ב-Workflow

המערכת מורכבת מ-5 שלבים (`Steps`) מרכזיים:

1. **`route_query` (שלב הנתב):**
   * **תפקיד:** מקבל את ה-`StartEvent` עם שאלת המשתמש ומנתח את כוונת השאילתה בעזרת בדיקת מילות מפתח סמנטיות.
   * **תוצאה:** מפיק `StructuredDataEvent` (עבור החלטות/כללים/אזהרות) או `QueryEvent` (עבור שאלה פתוחה).

2. **`process_structured_data` (שלב הנתונים המובנים):**
   * **תפקיד:** מופעל מ-`StructuredDataEvent`. שולף את הפריטים המתאימים מתוך קובץ ה-JSON (`project_data.json`), מחלץ את השדות הנדרשים ומעביר ל-LLM לניסוח תשובה קריאה.
   * **תוצאה:** מפיק `StopEvent` ומחזיר את התשובה הסופית.

3. **`retrieve` (שלב השליפה הוקטורית):**
   * **תפקיד:** מופעל מ-`QueryEvent`. פונה ל-`PineconeVectorStore` דרך מודל ה-Embedding ומחלץ את $k$ המסמכים הקרובים ביותר.
   * **תוצאה:** מפיק `RetrievalEvent` המכיל את הקטעים שנשלפו (`Nodes`).

4. **`validate` (שלב הדירוג והסינון):**
   * **תפקיד:** מופעל מ-`RetrievalEvent`. מעביר את הקטעים דרך ה-`CohereRerank` לקבלת ציון רלוונטיות מדויק, ומפעיל סף סינון (`score >= 0.0`).
   * **תוצאה:** מפיק `ValidationEvent` אם נמצא מידע תקין, או `StopEvent` עם הודעת סינון מבוקרת אם אין מידע רלוונטי.

5. **`synthesize` (שלב הניסוח הסופי):**
   * **תפקיד:** מופעל מ-`ValidationEvent`. משתמש ב-Synthesizer כדי לחבר את הקטעים המאומתים יחד עם השאילתה ליצירת תשובה רציפה.
   * **תוצאה:** מפיק `StopEvent` עם התשובה הסופית למשתמש.

## 🔬 4. דגשים מתקדמים לראיון (Advanced Architecture Concepts)

* **Handling Hallucinations (מניעת הזיות):** המערכת עושה שימוש כפול בסינון - גם ברמת ה-Reranker וגם ברמת ה-Threshold בשלב ה-Validate, כך שמידע לא רלוונטי נחסם לפני שהוא מגיע ל-LLM.
* **Token Optimization (אופטימיזציית משאבים):** על ידי סינון מקטעים לא רלוונטיים באמצעות ה-Reranker, חוסכים שליחת טוקנים מיותרים ל-LLM, מה שמפחית עלויות ומקצר את זמן התגובה (Latency).
* **Asynchronous Execution (הרצה אסינכרונית):** ה-Workflow מנוהל בצורה אסינכרונית (`async/await`), מה שמאפשר טיפול בבקשות במקביל ומניעת חסימה של שרת ה-UI.
* **Observability & Visualization:** ה-Workflow תומך בהפקת מפת אירועים אינטראקטיבית ב-HTML (`draw_all_possible_flows`), מה שמאפשר תחקור ויזואלי של כל השלבים במערכת.
